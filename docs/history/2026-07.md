# Auth probe: forward remembered doctors in on a slow/failed `/me` (2026-07-09)

Regression fix for `5038452` (the branded-splash-replaces-white-screen change),
`d3a2b94`. Two coupled symptoms, one broken path: a valid „Запомни ме" session was
forced to re-login, and the branded splash read as a pre-form gate.

- **Root cause.** In `app/app/login/page.tsx` + `app/signup/page.tsx` the auto-forward
  probe's new `PROBE_TIMEOUT_MS = 8000` timeout AND its network-failure/5xx `.catch`
  both called `setProbeFailed(true)` — flipping `probing` false, which renders the
  **login form**. The backend `/me` (`buildMeBody`) runs `authMiddleware` (synchronous
  `jwt.verify` → fast 401) *before* several sequential Supabase round-trips; a cold
  Railway `/me` exceeds 8s, the timeout fires, and a doctor holding a valid 30-day
  token is bounced to the form.
- **Fix.** Only a real **401** now clears the session (`clearSession()` +
  `setProbeFailed(true)` + `return`) and renders the form. **Timeout / network error /
  5xx** now `router.replace("/app/new-visit")` — forward into the app on the
  still-valid stored token. Safe because a timeout implies auth already passed (the
  slowness is the post-auth DB work), and the workspace shell
  (`app/(workspace)/layout.tsx`) gates on `getSession()` alone, so the doctor lands on
  the cached `doctor` profile — no white screen, no bounce.
- **Loader's correct home.** New `authenticating` flag, set only on submit SUCCESS
  (after `setSession`), drives `AuthBootSplash` via `if (probing || authenticating)`.
  The auto-forward effect early-returns while `authenticating` is true (deps include
  it) so the post-submit re-render can't fire a redundant probe. An un-remembered
  doctor (no token) still gets the form immediately — `probing` and `authenticating`
  are both false.
- **Untouched (verified by a fresh-context adversarial review).** `clearSession` (S4)
  still wipes `tuber_auth` from both stores + the PHI keys; sidebar/settings logout
  call sites unchanged; `setSession`/`getSession` remember-me persistence unchanged. A
  401 never reaches a forward path.
- **By-design tradeoff.** If the backend were slow enough to 401 only *after* 8s, the
  timeout forwards that dead token in and the workspace's own `/me` handles it. Relies
  on real 401s returning fast (synchronous JWT verify) — true for the cold-start case.
- **Verified live** (backend down = network-fail branch): no token → form instantly;
  stored token + failed probe → straight into `/app/new-visit`, workspace rendered from
  cached profile, no white screen; logout → both stores cleared → form. Gates: `build`
  + `tsc --noEmit` clean, lint zero new. Left for QA (needs live backend): real 401 →
  form, the post-submit splash on a real login, and the >8s throttle timeout branch.
